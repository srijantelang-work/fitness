# Phase 2: Database and Supabase Setup

In this phase, we will configure our Supabase PostgreSQL database to store user profiles, their generated workout plans, and daily logs.

## 1. Accessing the SQL Editor
1. Log in to your project dashboard on [Supabase](https://supabase.com/).
2. On the left sidebar, click on **SQL Editor**.
3. Click exactly on **New query**.

## 2. Table Schemas

Copy and paste the following SQL commands into the editor and click **Run**. This will create the 4 core tables we need.

```sql
-- Enable the UUID extension if not already enabled
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 1. Users Table (Extended from Auth)
-- This table automatically links to Supabase's built-in authentication users.
CREATE TABLE users (
  id UUID PRIMARY KEY REFERENCES auth.users ON DELETE CASCADE,
  name TEXT,
  age INT,
  weight_kg DECIMAL,
  height_cm DECIMAL,
  fitness_goals TEXT[],
  experience_level TEXT CHECK (experience_level IN ('beginner', 'intermediate', 'advanced')),
  health_conditions TEXT[],
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);

-- 2. Workout Plans Table
-- Stores the structured JSON plan generated by Gemini.
CREATE TABLE workout_plans (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  plan_name TEXT,
  target_goal TEXT,
  is_active BOOLEAN DEFAULT true,
  plan_json JSONB,             -- Example: {"day_1": {"focus": "legs", "exercises": [...]}}
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);

-- 3. Daily Workout Logs Table
-- Every time the user works out, a row is added here.
CREATE TABLE workout_logs (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  workout_date DATE DEFAULT CURRENT_DATE,
  exercises_completed JSONB,   -- Example: [{"name": "Squat", "sets": 3, "reps": 10, "weight": 100}]
  duration_minutes INT,
  calories_burned INT,
  agent_notes TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);

-- 4. Chat Logs Table
-- Long-term memory for the OpenClaw agent.
CREATE TABLE chat_logs (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  message TEXT NOT NULL,
  role TEXT CHECK (role IN ('user', 'agent', 'system', 'tool')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);
```

## 3. Setting Up Row Level Security (RLS)

Because we will query Supabase directly from the Next.js frontend, we must protect the data so users can only see their own workouts.

Run the following SQL query in the Supabase SQL editor:

```sql
-- Enable RLS on all tables
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE workout_plans ENABLE ROW LEVEL SECURITY;
ALTER TABLE workout_logs ENABLE ROW LEVEL SECURITY;
ALTER TABLE chat_logs ENABLE ROW LEVEL SECURITY;

-- Create Policies for 'users'
CREATE POLICY "Users can view their own profile" 
ON users FOR SELECT USING (auth.uid() = id);

CREATE POLICY "Users can update their own profile" 
ON users FOR UPDATE USING (auth.uid() = id);

CREATE POLICY "Users can insert their own profile" 
ON users FOR INSERT WITH CHECK (auth.uid() = id);

-- Create Policies for 'workout_plans'
CREATE POLICY "Users manage their own plans" 
ON workout_plans FOR ALL USING (auth.uid() = user_id);

-- Create Policies for 'workout_logs'
CREATE POLICY "Users manage their own logs" 
ON workout_logs FOR ALL USING (auth.uid() = user_id);

-- Create Policies for 'chat_logs'
CREATE POLICY "Users manage their own chat logs" 
ON chat_logs FOR ALL USING (auth.uid() = user_id);
```

## 4. Supabase Authentication Setup (Optional tweaks)
1. Go to **Authentication -> Providers** in the Supabase sidebar.
2. Email authentication is enabled by default. 
3. If you want, you can disable "Confirm email" for faster local development/testing (Authentication -> Providers -> Email -> Toggle off "Confirm email").
